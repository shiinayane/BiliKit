# CDN 镜像测速指南

给 [`scripts/cdn-pick.user.js`](../scripts/cdn-pick.user.js) 选最优镜像。核心原则：**延迟会骗人，只认真实下载吞吐**；单样本会骗人，**多样本聚合**才靠谱。

## 为什么要多测

吞吐随两个维度剧烈波动：

- **时段**：中国晚高峰跨境链路拥塞，同一镜像可能从 40Mbps 掉到 0。
- **冷热**：冷门/新视频没被边缘缓存，每个镜像都得**回源**拉，吞吐普遍腰斩，海外镜像甚至直接回源失败（HTTP 514）。卡顿基本都发生在这种内容上。

所以要在**不同时段**、用**冷热混合**的样本各测几轮，看谁的**中位吞吐高且成功率高**。

## 怎么选测试视频

凑 8~12 条，覆盖：

- **冷门**（重点，5~8 条）：几十到几百播放、最近几天刚发的。这类最考验回源，是你卡顿的真实场景。找法：冷门分区往下翻、看新发布、搜小众关键词。
- **热门**（2~3 条）：百万播放的老视频，作对照（边缘都缓存好了，谁都快）。
- 尽量分散在不同 UP / 分区（回源路径不同）。

## 收集 URL

每条视频：打开播放页 → 等它开始播 → 控制台跑这句，复制输出那条长 URL：

```js
performance.getEntriesByType('resource').map(e=>e.name)
  .filter(u=>/\.m4s/.test(u) && /bilivideo\.com/.test(u) && /upsig=/.test(u)).slice(-1)[0]
```

每条占一行，存进 `urls.txt`（`#` 开头的行会被忽略，可加注释标冷热）。

> URL 带 `deadline`（几小时有效），收齐后**尽快**跑完。装着 cdn-pick 时抓到的是 mirrorhwb 主机的地址，签名通用、可被测脚本换到所有镜像，没问题。

## 跑批量测速

```bash
bash test/cdnbatch.sh urls.txt
```

输出按中位吞吐排序，每个镜像给：**中位 Mbps / 均值 / 最低 / 成功率**。

```
镜像                             中位Mbps     均值     最低   成功/总
upos-sz-mirrorhwb.bilivideo.com     35.2     33.1     8.4    12/12
upos-sz-mirrorhw.bilivideo.com      31.0     29.5     5.1    12/12
upos-sz-mirrorcosov.bilivideo.com   12.3     14.8     0.0     9/12   ← 海外，冷门常失败
...
```

- **流量**：每探测默认下 10MB，约 `样本数 × 100MB`。想更准：`CHUNK=20971520 bash test/cdnbatch.sh urls.txt`（每探测 20MB）。
- **单条快测**（不聚合）：`bash test/cdntest.sh '<单条URL>'`。

## 怎么读、怎么定

1. **优先看中位 + 成功率**，别看均值（一次好窗口会把均值拉高骗你）。
2. **成功率 < 100% 的直接淘汰**——它会在关键时刻掉链子（海外镜像在冷门样本上常 <100%）。
3. 在剩下「成功率满 + 中位高」的里挑第一名，填进 cdn-pick 的 `TARGET_HOST`；第二三名填 `BACKUP_HOSTS`。
4. **多时段重复**：白天测一轮、中国晚高峰（约 21:00–24:00 北京时间）再测一轮。要是某镜像晚高峰崩了、另一个稳，按晚高峰的结果选——那才是你最需要它扛住的时候。

## 当前结论（2026-06，日本，待你扩样本验证）

大陆华为系（`mirrorhwb`/`mirrorhw`）冷热两头最稳；海外镜像（`cosov`/`aliov`）延迟低但冷门内容回源失败。cdn-pick 默认即 `mirrorhwb`，备 `mirrorhw`/`upcdnbda2`。多测后如有更优解，改 `TARGET_HOST` 一行即可。
